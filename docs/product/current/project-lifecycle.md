# Project Lifecycle

## Lifecycle Goals

The project workflow must support both initial applications and later owner-submitted edits without allowing unreviewed content to go live.

## Suggested MVP States

1. Draft
2. Pending review
3. Revision requested
4. Approved and unpublished
5. Published and awaiting seed
6. Sown
7. Completed
8. Archived

## Change Approval Rule

After a project has been submitted, Creative Evangelists should not edit the live record directly.

Instead:

1. the owner proposes a change
2. the proposed change enters review
3. a gatekeeper approves, rejects, or requests revision
4. only approved changes are applied to the live version

This same rule applies to edits after a project is already published.

## Seed Rule For MVP

- one project can have at most one seed
- one seed belongs to one project
- a project is either unsown or sown

This simplifies both public messaging and internal workflow for MVP.

## Visibility

- draft and in-review items are private to relevant internal users and the Creative Evangelist
- published projects are public
- archived non-case-study items are internal only
- case studies are public

## Audit Expectations

The system should retain:

- who submitted a project
- who requested revisions
- who approved or rejected a submission or edit
- when a project became public
- when a project became sown
